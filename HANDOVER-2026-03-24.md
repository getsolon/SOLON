## What Was Built

In one session, Solon went from "idea" to working product prototype. The goal: **"OpenClaw, safely — with any model"** — managed hosting where customers get OpenClaw running in secure Docker sandboxes with any inference provider.

### Architecture

```
Browser (Chat UI)
  → Solon Dashboard (React SPA, embedded in Go binary)
    → Solon Gateway (auth, rate limiting, guardrails)
      → OpenClaw Agent (Docker sandbox, agent-api on port 18790)
        → Anthropic/OpenAI/NVIDIA (customer's API keys)
```

### How OpenClaw Integration Works

1. `solon openclaw` or dashboard "Launch OpenClaw" button calls `EnsureOpenClaw()`
2. Manager builds a Docker image with OpenClaw pre-installed (commits container)
3. Container starts with a shell script that:
   - Starts OpenClaw gateway on loopback (`--auth none`, safe)
   - Starts an HTTP agent-api on `0.0.0.0:18790`
4. Agent API receives `POST /send {message}` and calls `openclaw agent --agent main --message "..." --json`
5. OpenClaw agent connects to its own gateway locally (gets admin scopes via self-pairing)
6. Agent processes the message (can use tools, browse web, execute code)
7. Returns JSON response with `result.payloads[0].text`
8. Solon's `POST /api/v1/openclaw/send` proxies this to the browser

### Security Model

- **Solon Gateway**: mandatory API key auth, rate limiting, guardrails (injection detection)
- **Docker Sandbox**: per-container isolation, cap-drop ALL, no-new-privileges
- **Tiered Networks**: Tier 1 = no internet (internal network), Tier 2+ = bridge with outbound
- **OpenClaw Gateway**: runs on loopback inside container, never exposed
- **Agent API**: accessible only via Docker bridge network, Solon proxies with auth
- **UFW**: port 8420 allowed from Docker subnet (172.16.0.0/12)
