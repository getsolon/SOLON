# Solon V1.2 — Production Pipeline Support

## Context

First real user feedback from a multi-agent TypeScript pipeline team (8 phases, mix of Opus/Sonnet/Haiku). They tested the relay successfully and want to offload low-complexity agents (chat, proofreading, fact extraction) to Solon/local models while keeping Claude for hard creative work.

**What already works for them:** OpenAI-compatible API, relay zero-config, key auth with model restrictions, HuggingFace direct pull.

**What they need to go to production:** see below.

---

## Tier 1: Easy Wins (implement first, hours each)

### 1.1 Model Warmup API
**Request:** Runtime API to preload a model before it's needed.
**Why:** Cold-loading a 27B model mid-pipeline causes timeouts. `--preload` only works at startup.
**Implementation:** Add `POST /api/v1/models/warmup` endpoint that calls `engine.ensureLoaded(model)` in background. Returns immediately with `{"status": "warming"}`. Non-blocking.
**Files:** `internal/gateway/gateway.go` (new handler + route)

### 1.2 Health Check with Model Readiness
**Request:** Health endpoint that shows which models are loaded and ready.
**Why:** Pipeline can poll at startup and wait until required model is warm.
**Implementation:** Merge `/api/v1/models/loaded` data into `/api/v1/health` response. Add `loaded_models` array.
**Files:** `internal/gateway/gateway.go` (update `handleHealth`)

### 1.3 Relay Latency Headers
**Request:** `X-Solon-Inference-Ms` and `X-Solon-Relay-Ms` response headers.
**Why:** Distinguish model inference time from network overhead to decide model size.
**Implementation:**
- Go side: already tracks `time.Since(start)` — add as response header
- Relay Worker: timestamp on request arrival, calculate relay overhead, add header
**Files:** `internal/gateway/gateway.go` (add header in handlers), `relay/src/index.ts` (add timing), `internal/relay/client.go` (pass through headers)

### 1.4 Token Budget Per Key
**Request:** `max_tokens_per_day` limit per API key.
**Why:** Prevent CI smoke tests from saturating GPU while users are generating.
**Implementation:** Add `max_tokens_per_day INTEGER` column to `api_keys`. Track daily usage in `requests` table (already logged). Check in auth middleware after key validation.
**Files:** `internal/storage/db.go` (migration), `internal/storage/keys.go` (new field), `internal/gateway/auth.go` (budget check query)

## Tier 2: High Value, Moderate Effort (days each)

### 2.1 Structured JSON Output (`response_format`)
**Request:** OpenAI-compatible `response_format: { type: "json_object" }` parameter.
**Why:** Pipeline validates every LLM output through Zod schemas. Needs guaranteed parseable JSON.
**Implementation:** llama.cpp supports GBNF grammars for constrained output. Map `response_format.type == "json_object"` to a JSON grammar. Pass grammar to llama.cpp backend.
**Complexity:** Medium — need to generate/embed a JSON GBNF grammar, wire through engine → backend.
**Files:** `internal/inference/engine.go` (pass format), `internal/inference/backends/llamacpp.go` (apply grammar), new `internal/inference/grammars/json.gbnf`

### 2.2 Streaming Token Usage
**Request:** Final SSE chunk should include token usage when `stream_options: {include_usage: true}`.
**Why:** They track per-phase token metrics and lose data when streaming.
**Implementation:** OpenAI spec: when `stream_options.include_usage` is set, the final chunk includes a `usage` field. Count tokens during streaming, emit in the `[DONE]` or final chunk.
**Files:** `internal/gateway/gateway.go` (update `handleChatCompletionsStream`), `internal/inference/engine.go` (expose token count from stream)

### 2.3 Task-Type Tags in Model Catalog
**Request:** Catalog entries tagged with task suitability: "good for code", "strong at German", "fast for classification".
**Why:** They have agents with different needs and don't want to A/B test every model.
**Implementation:** Add `good_for` string array to catalog.json entries. Curate based on public benchmarks. Surface in dashboard and `/api/v1/models/catalog` response.
**Files:** `internal/models/catalog.json` (add tags), `internal/models/catalog.go` (update struct), `dashboard/src/pages/instance/Models.tsx` (show tags)

## Tier 3: Bigger Features (defer to V1.3+)

### 3.1 Function Calling / Tool Use
**What:** OpenAI-compatible `tools` and `tool_choice` parameters.
**Why not now:** Requires tool-use prompt templates per model family (Llama uses `<|python_tag|>`, Mistral uses `[TOOL_CALLS]`, etc.). Each model handles it differently. Wait until they actually need it.
**When:** When their pipeline moves to agentic workflows.

### 3.2 Batch / Multi-Request Endpoint
**What:** `POST /v1/batch` accepting N requests, returning N responses.
**Why not now:** They can parallelize client-side with Promise.all. The relay adds ~50ms per request — batching saves round-trips but doesn't reduce inference time.
**When:** When relay latency becomes a measurable bottleneck.

### 3.3 Request Priority / Queuing
**What:** `"priority": "high" | "low"` hint on requests. High-priority requests jump the queue.
**Why not now:** Only matters with multiple concurrent users on one instance. Single-user pipeline doesn't need it.
**When:** When they have multiple users sharing one Solon instance.

```
Session 1: Tier 1 (easy wins)
  1.1 Model warmup API           ~1 hour
  1.2 Health with model readiness ~30 min
  1.3 Relay latency headers       ~1 hour
  1.4 Token budget per key        ~2 hours

Session 2: Tier 2 (high value)
  2.2 Streaming token usage       ~2 hours
  2.3 Task-type tags in catalog   ~1 hour (mostly curation)
  2.1 JSON mode / response_format ~4 hours (grammar integration)
```

Tier 1 can all be done in one session. Tier 2.2 and 2.3 are quick. Tier 2.1 (JSON mode) is the most impactful but needs grammar work.
